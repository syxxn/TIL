# **HTTP 특징**

1. Connectionless(비연결지향) : 클라이언트와 서버가 요청과 응답을 한 번 주고 받으면 연결을 끊어버리는 특징

   > 클라이언트가 request를 서버로 보내면 서버는 클라이언트가 보낸 request에 맞게 response를 보내고 연결을 끊음

2. Stateless(상태정보유지 안함) : 위처럼 요청과 응답으로 통신이 끝난다면 상태정보를 유지하지 않는 특징

   > ex) 메인페이지에서 로그인을 하고 다른 페이지로 넘어가면 다시 로그인을 해야함



**HTTP의 비연결성(클라이언트 요청에 대해 서버가 응답을 마치면 맺었던 연결을 끊어버리는 성질)으로 인해 매번 새로운 연결을 시도/ 해제의 과정을 거쳐야 하는 것을 막기 위해 사용.**

> 쿠키, 세션, JWT를 이용함으로써 사용자 인증에 대한 정보를 유지함

# 쿠키

**COOKIE (쿠키)**: 클라이언트에 저장되는 Key와 Value로 이루어진 작은 데이터파일. 

> 인증 유효 시간을 설정할 수 있고, 유효시간 안에는 클라이언트가 종료 되어도 인증이 유지됨.

> 서버와 요청 응답으로 인해 쿠키가 저장되면 다음 요청은 쿠키에 담긴 정보를 이용해 참조한다



### 쿠키 동작 방식

1. 클라이언트가 로그인 요청
2. 서버에서 쿠키 파일을 생성하여 클라이언트로 전달
3. 클라이언트가 서버로 요청을 보낼 때 쿠키를 전송
4. 쿠키를 이용하여 유저 인증 진행



#### 특징

+ 장점 : 서버 공간 절약
+ 단점 : <u>보안 취약</u>, 쿠키 용량 제한, 브라우저 공유 불가능, 쿠키의 사이즈가 커질수록 네트워크 부하

> 사용자 인증에 대한 정보를 모두 클라이언트가 가지고 있게되므로 http 요청을 탈취당할 경우 사용자 정보가 유출된다. 
>
> -> 그래서 쿠키자체는 보안과는 큰 관련이 없는 장바구니 혹은 자동로그인 설정 등에 이용함

# 세션

**세션**: 세션은 쿠키를 기반으로 하지만 쿠키와 다른 점은 서버에 저장하여 관리한다는 점이다

> 서버에서는 클라이언트를 구별하기 위해 각각의 세션ID를 클라이언트마다 부여하게 되며 클라이언트가 종료되기 전까지 유지한다





### 세션 동작 방식

1. 클라이언트가 로그인 요청
2. 서버에서는 클라이언트에게 고유한 세션 ID를 부여하고 세션 저장소에 저장한 후 클라이언트에게 발급
3. 클라이언트는 서버에서 발급받은 세션ID를 쿠키에 저장
4.  요청을 보낼 때마다 쿠키를 이용하여 세션 ID 값 전달
5. 서버는 쿠키에 담겨있는 세션ID와 세션 저장소에 있는 정보와 대조한 후 데이터를 가져옴



#### 세션의 특징

+ 세션 아이디는 브라우저 단위로 저장되고 브라우저 종료시 소멸됨
+ 로그아웃하면 새로운 사용자로 인식해서 새로운 세션이 생성된다



#### 장단점

+ 장점 : 쿠키보다 보안 향상, 용량 제한 X, 브라우저 의존도 낮음, 네트워크 부하 낮음
+ 단점 : 사용자가 많아질수록 서버 부하 증가

# JWT

Jason Web Token : Json 포맷을 이용하여 사용자에 대한 속성을 저장하는 Claim 기반의 Web Token

> Json을 사용하며 가볍고 안전성 있게 전달해 줌

>  **토큰기반 인증은 stateless(무상태) 서버에 적합함.** Stateless 서버는 상태유지를 하지 않음 
>
> > 유저의 인증 정보를 서버나 세션에 담아두지 않음



> Token이란 임시 인증키라 생각하면 쉬움(등본같은 느낌)
>
> 1. 많은 언어 지원
> 2. 자가 수용적
> 3. 전달 용이





### 토큰 동작 방식

> 웹서버에서 토큰을 서버에 전달할 때에는 HTTP 요청의 헤더에 토큰값을 포함시켜서 전달

1. 클라이언트의 로그인 요청

2. 서버측에서 해당 계정정보를 검증

3. 계정정보가 정확하다면, 서버측에서 클라이언트에게 *signed* 토큰을 발급

   > 여기서 signed 의 의미는 해당 토큰이 서버에서 정상적으로 발급된 토큰임을 증명하는 signature 를 지니고 있다는 것

   > 로그인을 새로 할 때마다 새로운 토큰을 발급해줌
   >
   > -> 토큰이 만료될 때까지 신경 안 씀

4. 클라이언트 측에서 전달받은 토큰을 저장해두고, 서버에 요청을 할 때 마다, 해당 토큰을 함께 서버에 전달합니다.

5. 서버는 토큰을 검증하고, 요청에 응답합니다.



#### 특징

+ 장점 : 세션보다 간편함, 쿠키 장점 + 세션 장점 , PC와 모바일 공유 가능
+ 단점 : 토큰 길이가 늘어날수록 네트워크 부하, 발급된 JWT는 삭제 불가

# HTTP

> Hyper Text Transfer Protocol : 글자 전환 프로토콜
>
> -> 인터넷에서 데이터를 주고받는 통신 규약

: Client의 요청(Request)이 있을 때만 서버가 응답(Response)하여 해당 정보를 전송하고 곧바로 연결을 종료하는 방식 (Server에서 Server로 요청 보낼 수 없음)

**HTTP 통신은 실시간 연결이 아닌, 필요한 경우에만 Server로 접근하는 콘텐츠 위주의 데이터를 사용할 때 용이함**



+ HTTP 통신의 특징

  + 클라이언트가 요청을 보내는 경우에만 서버가 응답하는 **단방향 통신**

  + 서버로부터 응답을 받은 후에는 연결이 바로 종료됨

  + 필요한 경우에만 서버로 요청을 보내는 상황에 유용

  + 요청을 보내 서버의 응답을 기다리는 애플리케이션(안드 or IOS)의 개발에 주로 사용

    -> 비용 및 유지보수 등 대부분의 방면에서 좋음



## HTTP에서 사용하는 통신 방법

### TCP 

> Transmission Control Protocol

: 인터넷상에서 데이터를 메세지의 형태로 보내기 위해 IP와 함께 사용하는 프로토콜 (연결)

-> 연속성보다 **신뢰성있는 전송이 중요할 때에 사용하는 프로토콜**



+ TCP 서버의 특징

  + 서버<u>소켓</u>은 연결만을 담당

    > 소켓(Socket)
    >
    > :  프로그램이 네트워크에서 데이터를 송수신할 수 있도록, 네트워크 환경에 연결할 수 있게 만들어진 연결부
    >
    > + 클라이언트 소켓 : 연결 요청을 보낼 때 사용되는 소켓
    >
    > + 서버소켓 : 요청을 받을 때 사용되는 소켓

  + 연결과정에서 반환된 클라이언트 소켓은 데이터의 송수신에 사용

  + 서버와 클라이언트는 1대 1로 연결된다

  + 스트림 전송으로 전송 데이터의 크기가 무제한

  + <u>패킷</u>에 대한 응답을 해야하기 때문에 성능(시간 지연, CPU소모) 낮음

    > 패킷(Packet)
    >
    > : 인터넷 내에서 데이터 전송을 효율적으로 하기 위해 데이터를 여러 개의 조각들로 나누어 전송하는데, 이 조각을 패킷이라 함.

  + Streaming 서비스에 불리(손실된 경우 재전송)



### UDP

> User Datagram Protocol

: 데이터를 <u>데이터그램</u> 단위로 처리하는 프로토콜 (비연결)

> 데이터그램(Datagram)
>
> : 독립적인 관계를 지니는 패킷

-> 신뢰성보다는**연속성이 중요한 서비스에서 사용**



+ UDP 특징
  + 비연결형 서비스로 데이터그램 방식 제공
  + 정보를 주고 받을 때 정보를 보내거나 받는다는 신호절차를 거치지 않음
  + UCP 헤더의 CheckSum 필드를 통해 최소한의 오류만 검출
  + 신뢰성이 낮으나 TCP보다 속도가 빠르다



+ UCP 서버의 특징
  + UCP에는 연결 자체가 없어서 서버 소켓과 클라이언트 소켓의 구분이 없음
  + IP를 기반으로 데이터를 전송
  + 서버와 클라이언트는 1대 1, 1대 N, N대 M으로 연결이 가능하다
  + 흐름제어가 없어서 패킷이 제대로 전송되었는지, 오류가 없는지 확인할 수 없다
  + 파일 전송과 같은 신뢰성이 필요한 서비스보다 성능이 중요시 되는 경우에 사용







## 요청 

**클라이언트 입장에서 생각하기** /서버입장

[CRUD]

- **GET** - Read

  : 클라이언트가 서버에게 URL에 해당하는 자료의 전송을 요청한다.

  > 서버한테서 정보 가져오기 / 요청한 URI의 정보를 검색하여 응답

- **POST** - Create

  : 클라이언트가 서버에서 처리할 수 있는 자료를 보낸다. 예를 들어, 게시판에 글을 쓸 때 클라이언트의 문서가 서버로 전송되어야 한다.

  > 서버에 정보 올리기 / 요청된 자원 생성

- **PUT** - Update

  : 클라이언트가 서버에게 지정한 URL에 지정한 데이터를 저장할 것을 요청한다.

  > 서버에 있는 데이터 전체 **수정** / 요청된 정보 업데이트

- **DELETE** - Delete

  : 클라이언트가 서버에게 지정한 URL의 정보를 제거할 것을 요청한다.

  > 서버에 삭제 요청 / 요청된 정보 삭제

____

+ PATCH: 클라이언트가 서버에게 지정한 URL의 데이터를 부분적으로 수정할 것을 요청한다.

  > 서버에 있는 데이터 일부 수정

- ~~CONNECT: 클라이언트가 특정 종류의 프록시 서버에게 연결을 요청한다.~~
- ~~OPTIONS: 해당 URL에서 지원하는 요청 메세지의 목록을 요청한다.~~

> Get와 Head 요청은 원칙적으로 이를 호출한다고 해서 서버 측의 데이터에 변화가 있으면 안된다.(safe method)

> Get, Head, Put, Delete는 동일한 요청이 한번 전송됐을 때와 여러 번 연속하여 전송되었을 때의 처리 결과가 동일해야 한다.(Idempotent Method : 멱등적인 메소드)



## HTTP 응답 코드

+ 1 _ _ : 정보 전달 / 요청을 받았고, 작업을 진행 중이라는 의미
+ 2 _ _ : 성공 / 이 작업을 성공적으로 받았고, 이해했으며, 받아들여졌다는 의미
  + 200 OK : 성공적으로 처리했을 때
  + 204 No Content : 성공적으로 처리했지만 컨텐츠를 제공하지는 않음
  + 206 Paratial Content : 컨텐츠의 일부분만 제공
+ 3 _ _ : 리다이렉션(redirection) / 이 요청을 완료하기 위해서는 리다이렉션(방향 지정)이 이루어져야 한다는 의미
+ 4 _ _ : 클라이언트 오류 / 요청이 올바르지 않다는 의미
  + 400 Bad Request : 요청 자체가 잘못됨
  + 403 Forbidden : 서버가 요청을 거부할 때 / 차단 혹은 권한 X
  + 404 Not Found : 찾는 리소스가 없음
  + 410 Gone : 찾는 리소스가 영원히 사라진 경우
+ 5 _ _ : 서버 오류 / 올바른 요청에 대해 서버가 응답할 수 없음
  + 500 Internal Server Error(내부 서버 에러) : 서버에 오류가 발생해 작업을 수행할 수 없을 때
  + 502 Bad Gateway : 게이트웨이가 연결된 서버로부터 잘못된 응답을 받았을 때 사용
  + 503 Service Temporarily Unavailable(일시적으로 서비스를 이용할 수 없음) : 서비스를 일시적으로 사용할 수 없을 때(주로 과부화)
  + 504 Gateway Timeout : 게이트웨이가 연결된 서버로부터 응답을 받을 수 없을 때 사용





## + SOCKET 통신

: Server와 Client가 특정 Port를 통해 연결을 성립하고 있어 실시간으로 양방향 통신을 하는 방식

**연결지향형 통신이기 때문에 실시간 통신이 필요한 경우에 자주 사용됨**

+ Socket 통신의 특징
  + 서버와 클라이언트가 계속 연결을 유지하는 **양방향 통신**
  + 서버와 클라이언트가 실시간으로 데이터를 주고받는 상황이 필요한 경우에 사용
  + 실시간 동영상 스트리밍이나 온라인 게임 등과 같은 경우에 자주 사용됨

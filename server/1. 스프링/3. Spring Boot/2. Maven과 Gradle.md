### MAVEN

자바용 프로젝트 관리도구

+ 라이브러리의 관리를 매우 용이하게 해준다.

+ 프로젝트의 작성부터 컴파일, 페트스 등 프로젝트 라이프사이클에 포함되는 각 테스트를 지원

+ war파일 기반의 배포용으로도 자주 사용됨

  > war : **WAR**란 **W**ebApplication **Ar**chive의 약자로 말그대로 웹 어플리케이션 저장소이며 웹 어플리케이션을 압축해 저장해 놓은 파일



빌드 도구로 메이븐을 이용하려면 `spring-boot-starter-parent` 프로젝트를 부모 프로젝트로 상속한다.

 그러면 플러그인의 디폴트 설정, 의존 라이브러리의 버전 정의, 자바 컴파일러 준수 레벨, 문자 코드를 이어받을 수 있다. 값을 지정하거나 덮어 쓰지 않는다면 값을 정의하지 않아도 미리 준비된 디폴트 값으로 설정된다



```xml
<!-- Ingerit defaults from Spring Boot -->
<parent>
	<groupId>org.springframework.boot</groupId>
	<artifactId>spring-boot-starter-parent</artifactId>
	<version>2.0.6.RELEASE</version>
</parent>
<!--xml 설정-->
```



### GRADLE



그레이들을 이용하는 경우 설정을 이어받는 부모 프로젝트가 존재하지 않으므로 스타터를 의존관계로 추가해야 한다. 그레이들 플러그인 `spring-boot-gradle-plugin`으로 실행 가능한 JAR 파일을 작성한다



```groovy
buildscript{
	ext{
		springBootVersion = "2.0.6.RELEASE"
		groovyVersion = "2.5.3"
	}
	repositories{
		jcenter()
	}
    dependencies{
    	classpath "org.springframework.boot:spring-boot-gradleplugin:${springBootVersion}"
	}
}


...
```



 

### **MAVEN과 GRADLE의 차이점**

| MAVEN                                                        | GRADLE                                                       |
| ------------------------------------------------------------ | ------------------------------------------------------------ |
| + 빌드를 쉽게 할 수 있도록 해줌<br />+ pom.xml을 이용한 정형화된 빌드 시스템<br />+ 뛰어난 프로젝트 정보 제공<br />+ 개발 가이드 라인 제공<br />+ 테스트 소스 코드를 별도의 병렬 소스 트리에 보관<br />+ 테스트 케이스 명명 규칙을 사용하여 테스트 위치 및 실행<br />+ 테스트 케이스에 환경을 설정하고 테스트 준비를 위해 빌드를 사용자 정의하지 않아도 됨<br />+ 새로운 기능을 쉽게 설치 할 수 있고 업데이트할 수 있음 | + 유연한 범용 빌드 도구<br />+ maven을 사용할 수 있는 변환 가능 컨벤션 프레임 워크<br />+ 멀티 프로젝트에 사용하기 좋음<br /><br />+ 빌드를 설명하는 풍부한 도메인 모델<br />+ maven과 lvy 레포지토리 완전 지원<br /> |



### **Gradle이 Maven보다 좋은점**

- Build라는 동적인 요소를 XML로 정의하기에는 어려운 부분이 많다.
- Maven은 설정 내용이 길어지고 가독성 떨어진다.
- 의존관계가 복잡한 프로젝트 설정하기에 부적절하다
- 상속구조를 이용한 멀티 모듈 구현이 가능하다.
- 특정 설정을 소수의 모듈에서 공유하기 위해서는 부모 프로젝트를 생성하여 상속하게 해야 함 (상속의 단점 생김)
- Gradle은 Groovy를 사용하기 때문에, 동적인 빌드는 Groovy 스크립트로 플러그인을 호출하거나 직접 코드를 짜면 된다.
- Configuration Injection 방식을 사용해서 공통 모듈을 상속해서 사용하는 단점을 커버했다. 설정 주입 시 프로젝트의 조건을 체크할 수 있어서 프로젝트별로 주입되는 설정을 다르게 할 수 있다.

+ 그레이들은 스크립트를 작성하는 빌드 도구이므로 작업을 자유롭게 작성할 수 있다
+ 멀이 프로젝트를 구성할 때 프로젝트에 대해 일괄로 설정하고 필요에 따라 개별적으로 설정할 수 있어 스크립트의 작성량이 메이븐보다 적다
+ 메이븐은 특수한 처리가 필요할 때 독자적인 플러그인으로 구현해야 하지만, 그레이들은 스크립트를 작성하는 것만으로도 대응 가능하다

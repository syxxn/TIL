# Monolith Architecture

> "좋은 아키텍처는 시스템이 모노리틱 구조로 태어나서 단일 파일로 배포되더라도, 이후에는 독립적으로 배포 가능한 단위들의 집합으로 성장하고, 또 독립적인 서비스나 마이크로 서비스 수준까지 성장할 수 있도록 만들어져야 한다. 또한 좋은 아키텍처라면 나중에 상황이 바뀌었을 때 이 진행 방향을 거꾸로 돌려 원래 형태인 모노리틱 구조로 되돌릴 수도 있어야 한다."
>
> By '클린 아키텍처' 도서

모노리스 아키텍처를 직역하면 '하나로 된 돌' + '건축물'이다.

아키텍처는 건축학에서 가져온 말로, 코드를 목적 혹은 기능에 맞게 분할한 형태를 말한다. 

**모노리스 아키텍처**는 하나의 애플리케이션에 모든 서비스가 다 들어가있는 설계 방법을 말하며, 서비스를 확장하면 그대로 불려진다.

> 하나의 '애플리케이션'이란?
>
> + 애플리케이션은 하나의 시스템(카카오톡)을 말하고, 서비스는 애플리케이션에 있는 기능(채팅)이다



#### 장점

+ 관리와 유지보수가 쉽다

+ 테스트, 그에 따른 배포 파이프라인 구성이 간단하다

+ 시스템 구조가 간결하고 빠른 개발이 가능하다

+ 인프라 구조 구축, 운용이 간단하다

  > 인프라는 서버가 돌아가는 환경을 말한다



#### 단점

+ 대부분 단일 기술 스택(하나의 언어)을 사용한다

+ 배포 프로세스가 모든 파일에 돌아간다 → 코드 한 줄만 바꿔도 다시 테스트 빌드 푸시

  > 머리부터 발끝까지 다 돌아감

+ 서비스 장애가 전체 시스템 장애로 확산될 수 있다



### MSA랑 Module Monolith를 비교해보자!

구조만으로 집에 비유를 해보면,

MSA는 단독 주택이고, MSA는 하나하나마다 하나의 애플리케이션이 된다.

modular Monolith는 아파트라고 생각할 수 있다. 아파트 한 건물 안에 여러 건물이 있지만 호실 각각이 다른 가구인 것처럼 **서비스마다 나눈 뒤 하나의 애플리케이션으로 묶어주는 것**을 모듈형 모노리스라고 한다. 모듈형 모노리스는 각각의 서비스마다 상호작용을 한다. 



modular Monolith를 사용하면 MSA로 전환이 쉽다.
